The Identity Problem: Why SHA-256 Fails for AI Agent Skills

The Myth of Integrity

You verify your npm packages. You check GPG signatures on Docker images. You trust nothing.

And yet, when you install an AI agent skill, you're probably just trusting the hash.

SHA-256: "This file hasn't been tampered with." That's useful. But it's not enough. Not even close.

What SHA-256 Actually Verifies

File: skill-bundle.tar.gz
SHA-256: a1b2c3d4e5f6...

This tells you:

  • ✅ The file bits are identical to what was published
  • ✅ No corruption during download

This tells you nothing about:

  • ❌ WHO published it
  • ❌ Whether that person is trustworthy
  • ❌ Whether the skill has been updated by an imposter
  • ❌ What the skill actually does when it runs

The Attack You've Never Heard Of

Here's how credential theft actually works now:

  1. Legitimate skill published — Developer publishes "code-review-helper" to a marketplace
  2. Attacker's turn — Attacker creates "code-review-helper-v2" with the same name, different code
  3. Search result gaming — New version appears as "updated" in search results
  4. You install it — Developer sees "update available," clicks install
  5. Your secrets are gone — New version exfiltrates environment variables on first run

The SHA-256 check passes. The file is valid, just not by who you think.

The Missing Link: Identity Chaining

What we actually need is this:

SignedSkill:
  ├── ContentHash: SHA-256(skill-bundle)
  ├── PublisherIdentity: cryptographic identity (not username)
  ├── Timestamp: when signed
  └── PermissionScope: what it claims to do

This creates a chain:

  • Integrity — SHA-256 (has the file changed?)
  • Identity — Who signed it? (can we verify they are who they claim?)
  • Non-repudiation — When was it signed? (can we detect replay/rollback?)

The Real Cost of Ignoring This

Every day developers integrate skills that can read their codebase, execute commands, access API keys, and have NO identity verification.

This is like installing random npm packages in 2015, before the ecosystem matured. We're in that same moment for AI agent skills. The difference? AI agents have far more privilege than a JavaScript package.

What's Next

The AI agent security ecosystem needs:

  1. Identity-first skill manifests — Identity verification as default, not opt-in
  2. Permission declarations — Skills explicitly declare what they can access
  3. Transparent versioning — Can't fake an "update"
  4. Incident response — When something goes wrong, know who to contact
