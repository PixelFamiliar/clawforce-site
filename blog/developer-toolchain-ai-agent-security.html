The Complete Developer Toolchain for AI Agent Security

We spent the last month building something specific: a security toolchain that meets developers where they already work. Not a dashboard. Not a platform. Five tools that plug into workflows you already have.

Here's what's available today, why each piece exists, and how they fit together.

The Problem with "Security Platforms"

Most security products want you to adopt their world. New dashboard. New login. New workflow. New mental model. The result? Developers run the tool once during setup, never again.

We built the opposite. Five tools, five integration points, zero new workflows:

1. The CLI: For Developers Who Think in Terminals

The CLI is the foundation. npm install -g clawforce and you're scanning. 22 vulnerability patterns across 6 categories. Zero external dependencies, zero configuration.

# Scan your current project
clawforce scan .

✓ ClawForce Scan Complete
Risk Score:  9/100 — PASS
Findings:    1 (medium: os.environ.get for KEY/TOKEN)

# JSON output for scripts and pipelines
clawforce scan . --json | jq '.risk_score'

# Strict mode: fail if risk > 20
clawforce scan . --threshold 20

Design decision: zero dependencies. The scanner itself is a single file. No npm ecosystem risk.

2. The GitHub Action: For Teams Who Ship

Three lines in a YAML file and every push gets scanned. Every PR gets a security check. The action outputs structured data you can use in downstream steps.

# .github/workflows/security.yml
- uses: PixelFamiliar/clawforce-scan-action@v1
  id: scan
  with:
    fail-threshold: '40'

# Use the outputs
- run: echo "Score: ${{ steps.scan.outputs.risk-score }}"

The action uploads a full JSON report as a build artifact. 30-day retention. Useful for audits, compliance, or just debugging why a build failed.

3. The VSCode Extension: For Catching Issues Early

Security on save. Every time you hit Ctrl+S, the extension scans the file and puts findings directly in the Problems panel. No context switch. No separate tool.

Features worth noting:

  • Status bar integration — current file's risk score at a glance, green/yellow/red
  • Workspace scan — scan all files with a progress bar
  • Configurable threshold — only show findings above your severity level
  • Cursor compatible — works in both VSCode and Cursor

4. The SDKs: For Runtime Trust Decisions

Sometimes you need to verify trust at runtime. Before your agent loads a skill. Before your platform lists a plugin. Before your marketplace approves a submission.

// Node.js — verify before loading
const { ClawForce } = require('@clawforce/sdk');
const client = new ClawForce();

const trust = await client.verify('some-skill');
if (!trust.trusted) {
    throw new Error(`Skill rejected: score ${trust.risk_score}`);
}

# Python — gate a marketplace
from clawforce import ClawForce
client = ClawForce()

result = client.scan('publisher/new-skill')
if result.risk_score > 40:
    reject_submission(result)

Both SDKs are zero-dependency. TypeScript types included in the Node.js version. Dataclass models in Python.

How They Work Together

The real value is the layered coverage:

  1. Develop — VSCode extension catches issues as you write code
  2. Commit — CLI gives a local check before you push
  3. Push — GitHub Action enforces standards on the team level
  4. Deploy — SDK verifies trust at runtime before loading skills
  5. Certify — Submit for manual review, get a badge and registry listing

Each layer catches what the previous layer missed. A developer might skip the local scan, but CI won't. CI might pass with a near-threshold score, but runtime verification catches the edge case.

What We're Not Building

No dashboard. No "security manager" role. No weekly reports that nobody reads. The toolchain produces findings where developers already look: terminal, editor, CI log, runtime exception.

If you want a dashboard, export the JSON. Pipe it to whatever you use. We're not building another thing to log into.
