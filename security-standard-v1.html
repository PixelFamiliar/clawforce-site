ClawForce Security Standard v1.0

v1.0.0 · March 2026

Specification for agent skill security certification · Published by Pixel Familiar

1. Purpose & Scope

This standard defines the security requirements, threat categories, risk scoring methodology, and certification tiers for OpenClaw agent skills evaluated by ClawForce. It applies to all skills submitted for scanning, verification, or certification through the ClawForce platform.

Goal: Provide a deterministic, transparent, and reproducible security evaluation framework that skill developers, platform operators, and end users can trust.

2. Threat Categories

ClawForce evaluates skills against six threat categories. Each category contains specific patterns matched via regex against source code.

2.1 Data Exfiltration

Detection of unauthorized outbound data transfer, including:

  • Base64 encoding combined with HTTP POST/PUT requests
  • DNS tunneling via dynamic hostname construction
  • Raw socket creation for arbitrary network communication
  • Outbound HTTP requests to external endpoints

2.2 Credential Harvesting

Detection of attempts to access stored credentials:

  • Reading .env or credentials.env files
  • Accessing sensitive environment variables (KEY, TOKEN, SECRET, PASSWORD)
  • System keychain/keyring access
  • Reading certificate and private key files (.pem, .key, .p12)

2.3 Code Obfuscation

Detection of techniques used to hide malicious behavior:

  • Base64 decode + exec chains
  • Hex-encoded byte sequences (potential shellcode)
  • Dynamic attribute access via getattr() on dunder methods
  • globals() function lookup for indirect execution
  • Dynamic __import__() calls
  • Runtime code compilation via compile()

2.4 Privilege Abuse

Detection of unauthorized system access:

  • Reading sensitive system files (/etc/passwd, /etc/shadow)
  • Glob patterns targeting sensitive directories (.ssh, .aws, .gnupg)
  • Process termination via os.kill()
  • Native library loading via ctypes
  • File deletion operations

2.5 Crypto Mining

Detection of cryptocurrency mining activity:

  • Mining-related terminology (stratum, hashrate, nonce, block_reward)
  • Known miner references (Monero, XMRig, CoinHive, CryptoNight)

2.6 Dangerous Built-ins

Detection of high-risk Python built-in usage:

  • exec() — arbitrary code execution
  • eval() — code injection risk
  • subprocess with shell=True — command injection

3. Secret Detection

In addition to pattern-based scanning, ClawForce detects hardcoded secrets matching these formats:

Type Pattern Example Prefix
OpenAI API Key sk-[a-zA-Z0-9]{20+} sk-proj-abc...
Google API Key AIza[a-zA-Z0-9\-_]{35+} AIzaSy...
Stripe Key (sk|pk)_(live|test)_[a-zA-Z0-9]{24+} sk_live_...
Telegram Bot Token \d{8,12}:AA[a-zA-Z0-9\-_]{33+} 12345678:AA...
AWS Access Key AKIA[A-Z0-9]{16} AKIAIOSFODNN...
GitHub PAT ghp_[a-zA-Z0-9]{36} ghp_xxxx...
High-entropy strings 30+ chars of [A-Za-z0-9+/=_-] (various)

4. Severity Levels

Severity Weight Description Examples
Critical 25 Direct security threat with high confidence exec+base64, hardcoded secrets, /etc/passwd access
High 10 Significant risk requiring review .env reading, keychain access, process killing
Medium 3 Moderate risk, context-dependent Environment variable access, raw sockets
Low 1 Informational, common patterns Outbound HTTP requests

5. Risk Scoring

Risk scores range from 0 to 100 and are computed as:

risk_score = min(100, Σ (severity_weight × confidence) for each finding)

Where:

  • severity_weight — the weight from the severity table above
  • confidence — a float from 0.0 to 1.0 indicating pattern match confidence

5.1 Risk Tiers

Tier Score Range Meaning
Pass 0 No security findings detected
Low Risk 1–15 Minor findings, generally safe
Medium Risk 16–40 Review recommended before deployment
High Risk 41–100 Significant security concerns — do not deploy without remediation

6. Certification Tiers

Tier Analysis Type Requirements
Scanned Static analysis only
  • All 22 vulnerability patterns checked
  • Secret detection applied
  • File extension filtering
  • Risk score computed
  • JSON report generated
Verified Static + Intent
  • All Scanned checks
  • SKILL.md claim vs code behavior comparison
  • Undeclared capability detection
  • Permission scope audit
  • Signed certificate issued
Certified Static + Intent + AI Review
  • All Verified checks
  • AI semantic code review
  • Supply chain deep analysis
  • Signed JSON certificate
  • Embeddable trust badge
  • Cryptographically signed certificate

7. Certificate Format

Certificates issued by ClawForce include:

{
  "skill_name": "example-skill",
  "version": "1.0.0",
  "certification_tier": "verified",
  "content_hash": "sha256:a1b2c3...",
  "risk_score": 3,
  "scan_timestamp": "2026-03-08T12:00:00Z",
  "scanner_version": "1.0.0",
  "issuer": "ClawForce · Pixel Familiar",
  "valid_for": "This specific version and content hash only"
}

Important: Certificates are valid only for the exact content hash at the time of scanning. Any code change invalidates the certificate and requires re-scanning.

8. File Coverage

8.1 Source Files Scanned

.py, .js, .ts, .jsx, .tsx, .mjs, .cjs, .sh, .bash, .zsh

8.2 Config Files Scanned

.env, .cfg, .ini, .yml, .yaml, .json, .toml

8.3 Excluded

.git, node_modules, __pycache__, .venv, venv, dist, build

9. Limitations

This standard explicitly acknowledges the following limitations:

  • Static analysis cannot detect all classes of vulnerabilities
  • Runtime behavior may differ from static analysis predictions
  • Pattern matching produces false positives — context matters
  • Zero-day vulnerabilities in dependencies are not detected
  • Tier 1 scans do not analyze imported libraries or transitive dependencies
  • Obfuscation techniques beyond the listed patterns may evade detection

10. Versioning & Updates

This standard uses semantic versioning. Changes to threat categories, severity weights, or risk scoring require a major version increment. Pattern additions within existing categories require a minor version increment.
